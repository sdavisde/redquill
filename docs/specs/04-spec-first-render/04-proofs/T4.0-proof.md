## Implementation summary

### 4.2 â€” pure helpers + flattened-row model

- `lineno_col_width` now walks every `Line.old_lineno`/`new_lineno` across
  all files (`flat_map` over hunks â†’ lines â†’ `Option<u32>` chain), takes the
  max, and returns its base-10 digit width via a small loop-based
  `digit_width` helper (avoids float `log10` rounding pitfalls). Empty input
  â†’ `0` â†’ `digit_width(0) == 1`.
- A private `Row` enum (`FileHeader(usize)`, `HunkHeader(usize, usize)`,
  `Placeholder(usize)`, `Line(usize, usize, usize)`) addresses rows by
  `(file, hunk, line)` index rather than pre-rendered text. `file_row_count` +
  `total_rows` give the flattened count (1 header + hunk-headers/lines, or 1
  header + 1 placeholder for binary/zero-hunk files â€” spec Â§6).
  `rows_in_range(files, range)` walks the structure with a running global
  index and **stops as soon as `range.end` is reached** (labeled-loop early
  break), so materializing the visible slice never walks past the requested
  window â€” this is the concrete mechanism behind "never build/style the
  whole buffer per frame" (FR-render-view-5, spec Â§6 large-diff). `total_rows`
  itself is computed once in `run`, not per frame.
- Long-line truncation: `styled_content_spans` operates on `content.chars()`
  (matching `changed_spans`' documented char-index convention), truncates to
  `max_width` chars with a trailing `â€¦` when the line doesn't fit, and splits
  the visible portion into styled runs so `changed_spans` regions get
  `emphasis_style` (`Modifier::REVERSED` over the base color) distinct from
  the rest of the line (FR-render-view-3/4).
- Gutter/line-number columns: `gutter_marker` (frozen, unchanged) + a new
  `lineno_field` (right-aligned, blank-padded to `lineno_width` when a side
  has no line number, e.g. a pure addition's old side).

### 4.3 â€” `run(files)` event loop

- Constructs `Keymap::default_map()` and `TerminalGuard::enter()?` once. All
  key behavior routes through `keymap.resolve(ev)` â€” the loop's `match`
  dispatches on the crossterm `Event` variant (`Key`/`Resize`/wildcard-ignore),
  never on `KeyCode` â€” no `match ev.code` behavior arms exist anywhere in
  `app.rs` (grep-verified: only `KeyCode::Char(c)` inside `describe_chord`'s
  *formatting* match, which is display, not behavior).
- Draws once before the loop, then blocks on `crossterm::event::read()`
  (FR-render-view-5: render-on-event, never a busy loop). A key event is
  filtered to `KeyEventKind::Press` (guards against double-processing if a
  terminal reports Repeat/Release â€” inert on typical Unix terminals where
  only `Press` is ever emitted without the kitty keyboard protocol enabled).
  `Resize` recomputes `viewport` and re-clamps `scroll`, then always redraws.
  Every other action mutation goes through `apply_action`, which returns
  whether state changed; the loop only calls `draw` again when `changed` is
  true (or on resize) â€” "only redraw on state change".
- `ScrollDown`/`ScrollUp` move Â±1 row via `clamp_scroll`; `HalfPageDown`/
  `HalfPageUp` move Â± `half_page(viewport)` = `(viewport / 2).max(1)`,
  computed fresh from the *current* `viewport` on every event (spec Â§9
  default: "half the current viewport height, recomputed per event").
  `Quit`/`QuitDiscard` set `should_quit`; every other `Action` variant is
  matched explicitly and does nothing (FR-render-keymap-3) â€” there is no
  wildcard `_ => {}` arm in `apply_action`'s action match, so adding a new
  `Action` variant later without also handling it here is a compile error,
  not a silently-dropped case.
- No `unwrap()`/`expect()` in the loop; `crossterm::event::read()` and
  `Terminal::draw()` propagate via `?` into `UiError`/`crate::ui::UiError`.

### 4.4 â€” styling + empty state

- `render_row` colors `Added` green, `Removed` red, `Context` the terminal's
  default (`Color::Reset`) â€” plain colors, no syntax highlighting
  (FR-render-view-3). `changed_spans` regions render with
  `base_style.add_modifier(Modifier::REVERSED)` â€” a distinct, visually
  "brighter/inverted" emphasis layered on the line's base color
  (FR-render-view-4).
- File header rows render bold; hunk header rows render cyan; binary/
  zero-hunk placeholder rows render the literal string `"binary / no textual
  diff"` in dark gray (spec Â§6's exact placeholder wording), never an empty
  gap.
- Empty diff (`app.files.is_empty()`): `render_frame` takes an early branch
  that renders an explicit empty-state `Paragraph` â€” `"No changes to review.
  Press {quit_hint} to quit."` â€” and returns, with NO special-case bypass of
  the TUI itself: `run`'s `TerminalGuard`/`Keymap`/event-loop wiring is
  identical whether `files` is empty or not; only the frame's *content*
  differs (FR-render-wire-2). `quit_hint` is produced by
  `keymap.chord_for(Action::Quit)` â†’ `describe_chord`, never a hardcoded
  `'q'`/`"q"` literal (FR-render-keymap-5) â€” grep-verified: `app.rs` contains
  no `KeyCode::Char('q')` or string literal `"q"` anywhere; `describe_chord`
  formats whatever chord it's handed generically (`Ctrl-` prefix + `{:?}`
  fallback for non-char codes).

### 4.6 â€” panic-trigger observation (FR-render-term-2)

- Added a hidden, doc-commented, env-var-gated panic trigger at the top of
  `render_frame` (the function `Terminal::draw`'s closure calls every frame):
  ```rust
  if std::env::var_os("REDQUILL_PANIC_TEST").is_some() {
      panic!("redquill: REDQUILL_PANIC_TEST fired mid-render (manual FR-render-term-2 check)");
  }
  ```
  This fires literally inside the `Terminal::draw` closure â€” i.e. while raw
  mode is enabled and the alternate screen is active â€” so a panic here is a
  faithful "panic mid-render" (not merely mid-setup or mid-teardown). It is
  inert unless the operator explicitly sets the env var; no normal `cargo
  run` invocation is affected.

**Why manual verification could not be completed in this environment:** this
agent has no controlling TTY (confirmed below), so `TerminalGuard::enter()`'s
`crossterm::terminal::enable_raw_mode()` call fails immediately with `Device
not configured (os error 6)` before the render loop â€” and therefore before
the panic trigger â€” is ever reached:

```
$ cargo run --            # stdin/stdout not a TTY in this sandbox
Error: terminal I/O error: Device not configured (os error 6)
Caused by:
    Device not configured (os error 6)
```

This IS still a useful observation in its own right: on a genuinely
non-interactive/non-TTY invocation, `run()` fails fast with a clean
`anyhow`-wrapped error and exits promptly â€” it never hangs in a busy loop and
never leaves a half-entered raw-mode/alt-screen state (nothing was entered
yet), for either an empty or non-empty diff (the failure happens at
`TerminalGuard::enter()`, before `files` is even inspected).
