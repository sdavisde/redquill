## Implementation summary (2.2–2.5)

- **2.2 `KeyChord::from_event`:** normalizes `ev.modifiers` to
  `ev.modifiers & KeyModifiers::CONTROL`, dropping every bit except
  `CONTROL`. Rationale (doc-commented on the fn): a shifted character key
  (`Q`, `?`) or `BackTab` already encodes "shifted" in `code`, but different
  terminals disagree on whether the `SHIFT` bit also gets set alongside it;
  keeping the bit would split one physical key press into two different
  hash keys depending on the terminal. `CONTROL` is the only modifier the
  README map uses independently of `code` (`Ctrl-d`/`Ctrl-u`), so it's the
  only bit preserved.
- **2.3 `Keymap::default_map()`:** binds every README row exactly —
  `j`/`k`, `Ctrl-d`/`Ctrl-u`, `]`/`[`, `Tab`/`BackTab`, `/`, `n`/`N`, `c`,
  `v`, `Char(' ')`, `s`, `K`, `a`, `?`, `q`/`Q` — plus the four arrow/page
  aliases (`Down`, `Up`, `PageDown`, `PageUp`) as extra entries onto the
  same actions as their vim counterparts. `gd`/`gr` are registered as
  placeholder **single-entry** no-op bindings on the chord's second
  character (`Char('d')` → `GotoDefinition`, `Char('r')` → `FindReferences`,
  both with `KeyModifiers::NONE`) — these do not collide with `Ctrl-d`
  (different modifiers) or with any other NONE-modifier binding. A doc
  comment directly above these two inserts explains the placeholder and
  that real `g`-prefixed two-key chord sequencing lands in Task 5, at which
  point these two entries (and only these two) change.
- **2.4 `resolve()`/`chord_for()`:** `resolve` normalizes the incoming
  event via `KeyChord::from_event` and does a single `HashMap::get`,
  defaulting to `Action::Noop` — pure data lookup, no `match` on `ev.code`.
  `chord_for` does a linear reverse scan (`.iter().find_map(...)`) for the
  first chord bound to the given action.
- **2.5 traceability:** `// FR-render-keymap-1` on `from_event` and
  `resolve`'s doc line (table-lookup mechanism), `// FR-render-keymap-2` on
  `default_map`'s doc line, `// FR-render-keymap-3` inline at each
  bound-but-deferred binding group, `// FR-render-keymap-4` on `resolve`'s
  `Noop` fallback, `// FR-render-keymap-5` on `chord_for`'s doc line, the
  module-level doc comment, and the `gd`/`gr` placeholder comment. The
  module doc comment states the one-line-rebind invariant so it doesn't
  need repeating at every call site.

## Deviations from literal task text

None functionally. One clarifying decision not spelled out verbatim in the
task/spec text: which single key stands in for each of the `gd`/`gr`
placeholder chords. The task says only "placeholder single-entry no-ops
(doc-comment)" without naming the stand-in key. Chose the chord's second
character (`d` for `GotoDefinition`, `r` for `FindReferences`) since it's
mnemonic-adjacent and doesn't collide with any other `NONE`-modifier
binding in the README map; documented inline as provisional and explicitly
flagged for replacement in Task 5's real chord-sequencing work.
